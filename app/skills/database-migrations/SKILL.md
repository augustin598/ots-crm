# Database & Migration Specialist — OTS CRM

## Context
- ORM: Drizzle ORM with libSQL (Turso)
- Schema: `src/lib/server/db/schema.ts`
- Migrations: `drizzle/` directory (~115 migrations)
- Naming: `NNNN_adjective_name.sql` (auto-generated by Drizzle)
- Journal: `drizzle/meta/_journal.json`
- Scale: ~50 tenants, single Turso writer, no replicas

## Core Design Principles
- Prefer small, composable migrations over large multi-purpose changes
- Keep schema evolution backward-compatible during rollout windows
- Separate schema changes (DDL) from data changes (backfills)
- Treat production data as messy, non-uniform, and historically inconsistent
- Prefer additive changes first, destructive cleanup later
- Assume migrations can be interrupted and must be resumable

---

## Must

### Before Writing a Migration
1. **Read current schema** — `src/lib/server/db/schema.ts` to understand existing tables
2. **Check last migration index** — read `drizzle/meta/_journal.json` to know the current idx
3. **Plan the change** — define exactly what tables/columns/indexes change
4. **Separate schema from data** — DDL changes and large backfills must NOT ship as one step

### Writing Schema Changes
5. **Add tenantId** — Every new data table MUST have `tenantId: text('tenant_id').references(() => tenant.id)` unless it's a global table (user, session, tenant, tenant_user)
6. **Use project types** — Match existing column helpers from schema.ts: `text()`, `integer()`, `real()`
7. **Add indexes** — Create indexes on foreign keys, tenantId, and frequently queried columns
8. **Use explicit defaults** — New columns must have a default or be nullable when old code may still write rows
9. **Backward compatibility** — New schema must work with currently running code during rollout window

### Expand-Contract Pattern (for breaking changes)
10. **Expand** — Add new column/table alongside the old one
11. **Dual-write** — App code writes to both old and new during transition
12. **Backfill** — Migrate existing data (as separate idempotent job)
13. **Verify** — Confirm data integrity
14. **Contract** — Remove old column/table in a later deploy

### Generating & Running Migrations
15. **Generate** — Run `bunx drizzle-kit generate` to create the SQL migration
16. **ONE statement per file** — If generated SQL has multiple statements, split into separate migration files. Turso executes each file as a single statement
17. **Verify journal** — Confirm `drizzle/meta/_journal.json` matches every .sql file. No orphan files, no missing entries
18. **Run locally** — `bun run db:migrate` against local DB first
19. **Verify on Turso** — After remote deploy, run `PRAGMA table_info(table_name)` to confirm columns exist
20. **Verify constraints** — Check foreign keys, nullability, uniqueness, and indexes after deploy

### Backfills
21. **Make backfills idempotent** — Must be safe to re-run without duplicating or corrupting data
22. **Batch by primary key** — For large tables, process in ranges/batches to avoid Turso lock timeouts
23. **Resumable** — Backfill must be stoppable and restartable from where it left off
24. **Treat seed data separately** — Bootstrap reference data must be deterministic and rerunnable

## Never
- Never drop, rename, or repurpose a live column in the same deploy where old code may still read/write it
- Never add a new NOT NULL column without a nullable/default phase and a backfill plan
- Never combine schema change + large data rewrite + behavior change in one migration
- Never hand-edit `_journal.json` to "make things pass" without proving live schema matches expected state
- Never use `IF NOT EXISTS` to hide drift that should fail loudly during review
- Never assume production data is clean enough for a one-shot transform
- Never fetch tenant-scoped rows by bare ID during migration scripts
- Never ship a migration that cannot be safely rerun or safely skipped after partial failure
- Never let cleanup run before compatibility is verified in production

---

## Failure Smells

### Blockers
- Migration drops, renames, or reuses a column/table in one deploy
- New `NOT NULL` field with no staged rollout or backfill
- Schema DDL and large data mutation mixed in one step
- Plan depends on editing migration journal manually after failure
- Migration introduces a constraint without proving existing rows satisfy it
- Rollback path missing for high-risk structural change

### Needs Review
- No production-like migration rehearsal exists
- Migration changes tenant keys, indexes, or FK behavior without post-deploy verification
- No written rollout sequence or forward-fix plan
- Migration size too large to review in one pass
- Backfill logic has no batch/range strategy
- Post-deploy validation described vaguely instead of concrete SQL queries

---

## Turso-Specific Rules
- **"Database is locked"** — NOT a code issue. Delete the DB pod in Kubernetes (Lens), StatefulSet recreates it
- **Write timeouts** — Same fix: pod deletion. Don't add retry logic for write locks
- **Single connection** — Turso uses a single writer. Don't run concurrent migrations
- **No transactions in migrations** — Turso doesn't support multi-statement transactions in migration files
- **Lock timeout** — Wrap migration scripts with connection timeout + backoff for transient lock errors

---

## Common Failure Modes
- Old app instances read pre-migration shape while new code writes new shape → partial breakage during rollout
- Backfill fails halfway and reruns incorrectly because not idempotent
- Drift between generated migrations, journal state, and live schema causes deploy failures
- Legacy rows violate new assumptions → migration succeeds structurally but fails functionally
- A seemingly harmless rename breaks serialized assumptions in jobs, exports, or API payloads
- Partially applied migration leaves DB in state neither old nor new code handles

---

## Review Checklist
- [ ] Is the change additive first?
- [ ] Can old code survive the intermediate schema?
- [ ] Is the backfill resumable and idempotent?
- [ ] Are tenant fields enforced?
- [ ] Are indexes and constraints validated after deploy?
- [ ] Is cleanup deferred until after verification?
- [ ] Are validation queries documented?
- [ ] Could this be retried safely without manual intervention?
- [ ] ONE statement per migration file?
- [ ] Journal matches SQL files?
