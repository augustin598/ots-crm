# Independent Senior Review — Task Module

## Top 10 Findings (ranked by production blast radius)

1. **[P0] Cross-Client Data Leak (IDOR) | `src/lib/remotes/tasks.remote.ts:160`**
   - Code: `where(and(eq(table.task.id, taskId), eq(table.task.tenantId, event.locals.tenant.id)))`
   - Risk: The `getTask` and `getTaskComments` remotes only verify the `tenantId`. A client user can view full details and all comment threads of *any* task belonging to *any other client* in the same agency by simply guessing or iterating a taskId.
   - Fix: Add a client-user check: `if (event.locals.isClientUser && task.clientId !== event.locals.client?.id) throw error(403, 'Unauthorized');`

2. **[P0] Insecure Attachment Paths (File Theft) | `src/lib/remotes/task-comments.remote.ts:192`**
   - Code: `attachments.map((a) => ({ commentId, path: a.path, ... }))`
   - Risk: `createTaskComment` accepts a raw `path` string for attachments. A malicious client can provide paths to sensitive files (invoices, contracts, or marketing materials) belonging to other tenants/clients. If they then "view" their own comment, the `getAttachmentUrl` will generate a presigned download link for that stolen file.
   - Fix: Never trust the `path` from the client. Validate that the provided path was recently generated by the user's own session or verify its ownership in the `storage` metadata before linking.

3. **[P0] Broken Comment/Activity Chronology | `src/lib/server/db/schema.ts:311`**
   - Code: `createdAt: timestamp('created_at', ...).default(sql`current_date`)`
   - Risk: SQLite `current_date` returns only `YYYY-MM-DD`. All comments and activities created on the same day will have an identical timestamp (`00:00:00`). Chronological ordering in `getTaskComments` and `getTaskActivities` will be effectively random, breaking conversation flow and audit trails in production.
   - Fix: Change default to `current_timestamp` for the `task_comment` and `task_activity` tables to ensure high-resolution temporal ordering.

4. **[P1] Dead Reactive Updates (Fresh Instance Bug) | `src/lib/components/client-task/client-task-comments.svelte:115`**
   - Code: `.updates(getTaskComments(taskId), getTaskActivities(taskId))`
   - Risk: Calling the remote function inside `.updates()` creates *new* query store instances. The `.refresh()` is triggered on these new instances, while the UI is listening to the `commentsQuery` defined via `$derived`.
   - Fix: Pass the existing reactive variables: `.updates(commentsQuery, activityQuery)`.

5. **[P1] Kanban Performance O(N²) / UI Flicker | `src/lib/components/task-kanban-board.svelte:102`**
   - Code: `for (let p = 1; p <= doneLoadedPages; p++) { queries.push(getCompletedTasks({...})) }`
   - Risk: Every time a user clicks "More" in the Done column, *all* previous pages are re-instantiated and re-fetched from the server. This causes massive UI flicker and exponentially increasing database load as the list grows.
   - Fix: Refactor the kanban to append results to a persistent local array rather than re-running the entire query chain for every pagination step.

6. **[P1] Description Rendering Mismatch | `src/lib/components/client-task/client-task-description.svelte:9`**
   - Code: `<pre ...>{description}</pre>`
   - Risk: Task descriptions are authored in a Rich Text Editor (TipTap) on the Admin side and stored as HTML. Using `{description}` inside a `<pre>` tag will display raw HTML tags (e.g., `<p><strong>...</strong></p>`) to the client instead of rendered text.
   - Fix: Use `{@html description}` and ensure the content is sanitized via the existing `sanitizeCommentHtml` logic.

7. **[P1] Client Portal Permission Bypass | `src/lib/remotes/tasks.remote.ts:600`**
   - Code: `export const updateTaskPosition = command(...)`
   - Risk: This remote lacks any `isClientUser` check. A client user can manually invoke this endpoint to move their own tasks to `done` or `cancelled` without agency approval, or move tasks they don't own if they have the ID.
   - Fix: Add `if (event.locals.isClientUser) throw error(403);` to `updateTaskPosition`. Clients should use specific guarded actions like `reopenTask`.

8. **[P2] Missing Critical Index | `src/lib/server/db/schema.ts:255`**
   - Code: `export const task = sqliteTable('task', { ... })`
   - Risk: There is no index on `clientId` in the `task` table. Since almost every query in the Client Portal filters by `clientId`, performance will degrade linearly with the total number of tasks in the tenant, eventually causing timeouts for large agencies.
   - Fix: Add `index('task_client_idx').on(t.clientId)` to the `task` table definition.

9. **[P2] ZIP Export Memory Pressure | `src/routes/client/[tenant]/(app)/invoices/google-ads/+page.svelte:315`**
   - Code: `const zip = new JSZip(); ... const blob = await res.blob(); zip.file(fileName, blob);`
   - Risk: JSZip processes everything in memory. If a client attempts to bulk-download a year's worth of high-res invoice PDFs, the browser tab will likely crash on mobile or low-memory devices.
   - Fix: Implement a limit on the number of files per ZIP (e.g., 20) or move ZIP generation to the server/worker.

10. **[P2] Race Condition in Status Changes | `src/lib/remotes/tasks.remote.ts:510`**
    - Code: `await db.update(table.task).set({ status: 'pending-approval' }).where(...)`
    - Risk: If two users (e.g., an Admin and a Client) act on a task simultaneously, the optimistic UI in `task-kanban-board.svelte` might show a successful move while the server update fails or applies a conflicting shift.
    - Fix: Use conditional updates with a `version` column or explicitly check the `rowsAffected` returned by the Drizzle update.

## Patterns I noticed (good and bad)
- **The "Updates" Pattern Divergence:** The `.updates(...)` pattern is powerful for multi-query invalidation, but it's being used inconsistently. In some components, it's correctly applied to local variables; in others (the new portal), it's calling factory functions again, creating orphan stores that refresh into a void.
- **Data Locality:** The decision to keep Admin and Client components separate (e.g., `client-task-description` vs `task-detail-body`) is a good maintenance hedge given how much more restricted the Client UI is. However, the duplicated "Schedule Meet" logic is already drifting, which will make adding new features (like Meet recording links) a two-step chore prone to bugs.
- **Svelte 5 Adoption:** Excellent use of `$derived.by` for complex grouping (like the Google Ads monthly spend). The reactive logic is generally clean and avoids the "prop drilling" hell common in older versions.

## What looks solid
- **Tenant Isolation Base:** The layout-level `locals.tenant` and `locals.client` injection is robust. The server-side gate in `(app)/+layout.server.ts` is a critical safety net that prevents broad access leaks.
- **UX Polish:** Features like the `due-blink` animation and the floating selection bar for ZIP downloads show a high level of care for the end-user experience.
- **Advanced KPI Logic:** The `calculateHealthScore` and `calculateCampaignFatigue` utilities are highly sophisticated and add genuine value to the agency's offering.

## What I'd ship vs. what I'd hold
- **Ship as-is:** Navigation, Dashboard stats, Team management, Marketing material grid.
- **Fix before next release:** **[P0]** IDOR in `getTask`, **[P0]** Insecure file paths in comments, **[P0]** Timestamp resolution in schema (critical for chat history).
- **Backlog:** ZIP memory optimization, indexing, unifying the Google Meet modal logic.
